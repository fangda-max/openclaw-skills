# 需求文档写作规范与转化指南

## 一、原始材料 → 文档章节的转化规则

### 1.1 会议纪要/录音文字稿 → 业务背景与痛点

**提取策略：**
- 关注参与者描述的**现有工作方式**和**抱怨/不满**
- 关注"我们现在是..."、"问题是..."、"希望能..."等句式
- 忽略闲聊、重复、技术细节讨论

**转化示例：**

原始材料（会议录音文字稿节选）：
```
发言人1: 我们诉求主要就两个，一个是跟刚咱刚才类似这种物理的这样的
一致性的比对，再一个就是验收报告和合同的这样的一致性的。
发言人2: 如果说后续...我建议是整体给办好的，因为你印章库你只做一个比，
你后续如果说涉及到可能还会这样。
发言人3: 目前的话就是一个公司是如果只有跟这个名称，一个名称就一个章，
现在全匹配的话...
发言人1: 但是不是会有多份合同？
```

转化为需求文档：
```
在项目实施、合同交付等场景中，进度证明材料、交付物等附件常包含印章
信息。为保障印章真实性、防止伪造或篡改，需提供印章稽核能力。

当前主要痛点：
- 效率低下：人工逐一核对印章耗时耗力
- 标准不一：不同人员判断标准各异
- 库管理缺失：同一公司可能有多枚不同类型印章，缺乏统一管理
```

**另一领域示例（电商售后）：**

原始材料：
```
客服主管: 现在退货流程太乱了，客户打电话来不知道退到哪个仓库。
运营: 是的，有些退的是总仓，有些退到前置仓，客服每次都要查。
客服主管: 而且审批也慢，有时候客户等三四天才能拿到退款。
```

转化为需求文档：
```
在售后服务场景中，退货退款是客户满意度的关键环节。

当前主要痛点：
- 流程混乱：退货目的仓库不清晰，客服需手动查询
- 审批效率低：退款审批周期长，影响客户体验
- 规则不透明：退货政策因渠道而异，缺乏统一管理
```

### 1.2 建设方案 → 范围与边界 + 总体流程

**提取策略：**
- "场景说明"部分 → 业务场景
- "建设内容"部分 → 范围与边界的"做什么"
- "实现方式"部分 → 总体业务流程

**转化示例：**

原始材料（建设方案节选）：
```
嵌入式印章稽核实现方式：
  印章库管理 → 印章列表、新增、调整、删除、查看
  嵌入业务环节中的印章识别/稽核 → 识别执行、结果展示、对比、忽略恢复

独立印章稽核工具实现方式：
  印章智能对比入口 → 上传、对比记录
  印章智能对比处理 → 选取、识别、对比结果、保存
```

转化为需求文档的"范围"章节：
```
建设内容分为三个层次：
第一层：基础数据层 – 建立标准印章信息库
第二层：核心功能层 – 构建多元化稽核应用（嵌入式+独立式）
第三层：支撑扩展层 – 确保系统可用、安全、可集成
```

**另一领域示例（审批流程系统）：**

原始材料：
```
建设目标：实现请假、报销、采购三类审批线上化
实现方式：
  统一审批中心 → 待办列表、已办列表、发起申请
  流程配置 → 审批节点、审批人规则、超时规则
```

转化为需求文档：
```
建设内容分为三个层次：
第一层：基础配置层 – 建立组织架构与审批规则库
第二层：核心业务层 – 构建统一审批引擎（请假/报销/采购）
第三层：支撑扩展层 – 消息通知、移动端适配、数据看板
```

### 1.3 功能说明书 → 系统功能（第四章）

**提取策略：**
- 需求编号表格 → 实现逻辑中的具体规则
- 数据流描述 → 接口交互和状态流转
- 数据模型(字段表) → 字段说明
- 页面结构示意 → 界面原型描述

**转化规则：**
功能说明书通常是技术视角，需要扩展为业务视角：

| 功能说明书中有的 | 需求文档中需要补充的 |
|------------------|----------------------|
| 需求编号列表 | 业务目标、业务场景 |
| 字段类型定义 | 字段的业务含义、校验规则 |
| 页面组件名称 | 操作入口的完整路径 |
| 数据接口 | 上下游业务影响 |
| 按钮和操作 | 权限控制、二次确认规则 |

**转化示例：**

原始材料（功能说明书）：
```
FR-SM-01-04: 生效状态操作：查看、编辑、稽核、失效
FR-SM-01-05: 待生效状态操作：查看、编辑、稽核、生效、删除
```

转化为需求文档：
```
【实现逻辑】：
操作列按钮权限与逻辑：
  生效印章：
  - 所有有权限用户可见【查看】【稽核】
  - 仅印章管理员可见【编辑】【失效】
  - 无【删除】按钮（生效印章严禁删除）

  待生效印章：
  - 所有有权限用户可见【查看】【稽核】
  - 记录创建者或印章管理员可见【编辑】【生效】【删除】
```

**另一领域示例（订单管理）：**

原始材料：
```
REQ-ORD-03: 待付款订单可取消
REQ-ORD-04: 已付款订单可申请退款
REQ-ORD-05: 已发货订单不可直接取消，需走退货流程
```

转化为需求文档：
```
【实现逻辑】：
操作列按钮权限与逻辑：
  "待付款"订单：
  - 买家可见【付款】【取消订单】
  - 卖家可见【查看】（无操作权限）

  "已付款"订单：
  - 买家可见【查看】【申请退款】
  - 卖家可见【查看】【发货】

  "已发货"订单：
  - 买家可见【查看】【确认收货】【申请退货】
  - 无【取消订单】按钮（已发货订单不可直接取消）
```

### 1.4 界面原型/截图 → 界面原型描述

**转化策略：**
- 图片无法嵌入markdown时，用结构化文字描述替代
- 按区域从上到下、从左到右描述
- 标注关键交互元素（按钮、输入框、表格列）

**描述格式：**
```
抽屉形式：从右侧弹出，宽度1740px
布局：
  上部：展示XXX信息（只读）
  中部：
    左栏（图1）：XXX内容
    中栏（图2）：XXX内容
    右栏：XXX结果面板
  底部：【保存】【关闭】按钮
```

### 1.5 现有系统需求文档 → 集成上下文

**提取策略：**
- 提取与本系统有交互的功能模块和流程节点
- 提取对接接口和数据字段
- 用于"范围与边界"中描述系统集成点

### 1.6 需求文档模板 → 文档格式规范

如果用户提供了公司内部的文档模板，需要与本Skill的默认模板进行融合。

**融合优先级规则（公司模板 > 默认模板）：**

| 维度 | 处理方式 |
|------|----------|
| 章节结构 | 以公司模板为准。若公司模板缺少某些章节，判断是否需要补充 |
| 章节命名 | 以公司模板的命名为准（如公司用"系统概述"而非"背景与建设目标"） |
| 表格格式 | 以公司模板的列定义为准 |
| A/B类分类 | 若公司模板有分类体系，沿用；若无，使用默认的A/B类分类 |
| 功能模块要素 | 合并两套要素——公司模板要求的 + 默认6要素中公司模板未覆盖的 |
| 填写样例 | 优先引用公司模板中的样例 |

**融合流程：**
1. 先完整读取公司模板，识别其章节结构和填写要求
2. 逐章对比默认模板，标记差异点
3. 对差异点：公司模板有明确要求的从其要求，公司模板未涉及的用默认模板补充
4. 在阶段二输出大纲时，向用户说明"以下章节按贵司模板结构，以下章节为默认补充"

## 二、写作风格规范

### 2.1 用词规范

**核心原则：同一概念全文只用一个词，不混用。**

以下为默认推荐用词。如用户公司有既定用词规范（如习惯用"创建"而非"新增"），以用户规范为准，但仍需保证全文一致。

**默认推荐用词表：**

| 概念 | 默认推荐 | 常见替代（择一即可） |
|------|----------|----------------------|
| 增 | 新增 | 创建、添加 |
| 改 | 编辑 | 修改、更新 |
| 查 | 查看 | 详情 |
| 删 | 删除 | 移除 |
| 启用 | 生效 | 启用、激活 |
| 停用 | 失效 | 停用、禁用 |
| 确认 | 提交 | 保存并提交 |
| 数据项 | 字段 | 属性 |
| 操作元素 | 按钮 | 操作项 |
| 浮层 | 弹窗 | 对话框 |
| 侧浮层 | 抽屉 | 侧边栏 |

**使用规则：**
1. 阶段一结束时确定本文档的用词表（默认或用户指定），后续全文遵循
2. 状态名称用引号标注：如"生效"、"待生效"、"失效"
3. 角色名称全文一致：首次出现时给出完整定义，后续全文使用同一名称

### 2.2 句式规范

**功能描述用动宾结构：**
- 正确：提供对XXX信息的全量查询与基础操作入口
- 错误：这个页面是用来查询XXX信息的

**操作步骤用主谓宾：**
- 正确：用户点击【查询】按钮，系统将查询条件作为参数向后台发起请求
- 错误：点击查询后查询数据

**条件判断用"若...则..."：**
- 正确：若系统中无匹配记录，则提示用户手动选择或新增
- 错误：如果没有就让用户自己加

### 2.3 层级规范

**章节层级不超过5级：**
```
一、（Heading 2）
  1.1（Heading 3）
    1.1.1（Heading 4）
      1.1.1.1（Heading 5）
        具体内容（正文）
```

**列表嵌套不超过3层**

### 2.4 表格规范

**字段说明表必须包含5列**：字段名、类型、必填、可编辑、说明

**权限矩阵必须包含4列**：角色、描述、核心操作权限、数据权限范围

**性能指标表必须包含5列**：大类、小类、类型(A类/B类，纯A类项目可省略此列)、指标项、要求描述

## 三、信息缺口补全策略

当原始材料缺少某类信息时，按以下策略处理：

### 3.1 可自动补全的（使用通用方案）

**非功能需求 - 性能（未提供时使用以下默认值）：**
- 页面首屏加载：≤2秒
- 核心事务操作响应：≤3秒
- 复杂查询响应：≤5秒
- AI端到端处理：≤10秒（简单场景≤5秒）
- 同步接口响应：95% ≤500ms
- AI推理超时：30秒

**非功能需求 - 安全（未提供时使用以下默认值）：**
- 传输加密：HTTPS (TLS 1.2+)
- 敏感信息脱敏：部分字符替换为*
- 密钥/密码存储：AES-256加密
- 接口认证：Token鉴权
- 审计日志留存：≥6个月

**分页默认值：** 支持每页10/20/50/100条

**操作确认默认规则：** 删除、失效、生效等状态变更操作默认需要二次确认弹窗

### 3.2 需要提醒用户但可暂用占位符的

**接口定义：**
- 标注"具体接口信息待研发梳理"
- 列出已知的接口名称和用途

**AI准确率指标：**
- 标注"建议基于实测数据确定，以下为参考值"
- 图像识别类：>95%
- 特征提取类：>90%
- 异常检测类：>85%

### 3.3 必须向用户提问的

以下信息无法自行推断，缺失时必须提问：

| 缺失信息 | 提问方式 |
|----------|----------|
| 核心对象的状态有哪些 | "请告诉我XXX对象有哪些状态？状态之间如何流转？" |
| 角色和权限 | "系统有哪些用户角色？每个角色能执行哪些操作？" |
| 做什么/不做什么的边界 | "以下功能是否在本期范围内？XXX、XXX" |
| 与现有系统的集成方式 | "需要与哪些现有系统对接？对接什么数据？" |
| 字段级别的业务规则 | "XXX字段是否有特殊的校验规则或计算逻辑？" |

## 四、质量检查清单

生成文档后，逐项检查：

### 4.1 结构检查
- [ ] 按规模选定的章节齐全（轻量3章/标准7章/完整7章+附录）
- [ ] 复杂模块包含"业务目标+场景+流程+功能模块"四级结构；简单模块至少包含"场景+功能模块"
- [ ] 每个页面/操作包含A类6要素；B类功能按需包含扩展要素
- [ ] 标准/完整项目附带结构化维度总表

### 4.2 一致性检查
- [ ] 角色名称全文统一
- [ ] 状态名称全文统一（用引号标注）
- [ ] 字段名称跨页面一致
- [ ] 按钮名称统一（用【】标注）
- [ ] 权限描述前后一致

### 4.3 完整性检查
- [ ] 状态流转闭环（每个状态有来源和去向）
- [ ] 所有提到的按钮都有对应逻辑
- [ ] 所有字段都有说明
- [ ] 边界说明完整（做什么+不做什么）

### 4.4 业务逻辑检查
- [ ] 数据来源清晰（每个字段的数据从哪来）
- [ ] 操作权限无矛盾
- [ ] 流程间数据依赖关系正确
- [ ] 异常/边界情况有处理说明
