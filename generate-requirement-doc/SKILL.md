# 系统功能需求说明书生成

## 适用场景

用户提供一份或多份原始需求材料，要求生成一份完整的系统功能需求说明书。

原始材料可能包括：
- 会议纪要 / 录音文字稿
- 建设方案 / 可行性报告
- 功能说明书 / 技术规格
- 界面原型 / UI截图 / PDF
- 现有系统需求文档
- 需求文档模板（公司内部格式规范）
- 用户口头描述

## 工作流程

按以下5个阶段执行。默认每阶段完成后向用户确认再进入下一阶段；若用户要求"快速模式"或材料充分且明确，可跳过中间确认连续生成。

### 阶段〇：项目预判（进入阶段一之前自动执行）

快速浏览材料后，判断两个维度：

**维度一：项目类型**
| 类型 | 判断条件 | 影响 |
|------|----------|------|
| 纯A类 | 材料中无AI/大模型/智能识别相关内容 | 跳过所有B类要素，信息提取跳过第10项 |
| 混合A+B类 | 既有传统功能又有AI功能 | 逐模块判断A/B类 |
| 纯B类 | 整体以AI能力为主 | 所有模块使用B类扩展要素 |

**维度二：文档规模**
| 规模 | 判断条件 | 章节策略 |
|------|----------|----------|
| 轻量 | ≤3个功能模块，无系统集成 | 裁剪为3章（概述+功能+非功能），跳过接口和权限独立章节 |
| 标准 | 4-10个功能模块 | 完整7章结构 |
| 完整 | >10个功能模块或涉及复杂集成 | 7章+附录（术语表、数据字典等） |

将判断结果告知用户确认后继续。

### 阶段一：材料分析与信息提取

1. **读取全部材料**：逐个读取用户提供的所有文件
   - **docx文件**：优先使用python-docx提取文本和表格；若环境无python-docx，提示用户执行 `pip install python-docx` 或要求用户另存为txt/pdf后提供
   - **pdf文件**：使用Read工具读取
   - **md/txt文件**：直接读取
   - **图片/截图**：描述可见内容，无法提取时要求用户文字补充

2. **信息分类提取**：从材料中识别并提取以下关键信息（第10项仅当项目类型含B类时提取）：

| 序号 | 信息类别 | 提取目标 |
|------|----------|----------|
| 1 | 业务背景 | 行业领域、当前流程、痛点问题 |
| 2 | 建设目标 | 系统要达成的3-5个核心目标 |
| 3 | 功能模块清单 | 需要建设的功能模块列表 |
| 4 | 用户角色 | 系统涉及的所有角色及其职责 |
| 5 | 核心对象与状态 | 业务对象的属性、状态、流转规则 |
| 6 | 页面与交互 | 页面布局、字段、按钮、操作逻辑 |
| 7 | 业务规则 | 校验规则、权限约束、唯一性约束 |
| 8 | 系统集成 | 需要对接的外部系统及接口 |
| 9 | 非功能需求 | 性能、安全、加密要求 |
| 10 | AI/异步能力 | 涉及的AI、大模型、批处理、异步任务能力（仅B类项目） |

3. **信息缺口检测**：对照以下检查清单，识别缺失信息：

**必须有（缺失则必须向用户提问）：**
- 业务背景与痛点
- 功能模块清单（至少知道要做什么）
- 核心对象的状态流转

**强烈建议有（缺失则提醒用户）：**
- 用户角色与权限定义
- 范围与边界（做什么/不做什么）
- 字段级别的详细定义

**可选（缺失则用通用方案补充）：**
- 非功能需求（用行业通用标准填充）
- 接口详细定义（标注"待研发梳理"）
- 审计日志要求（用通用模板）

4. **输出提取摘要**：向用户展示提取结果，确认理解是否正确，补全缺口后进入下一阶段

### 阶段二：文档骨架生成

1. 读取 [document-template.md](document-template.md) 获取标准文档结构
2. 根据"阶段〇"的规模判断，决定采用轻量/标准/完整结构（参考 document-template.md 中的章节必选/可选标注）
3. 根据提取的功能模块清单，确定第四章的具体子章节结构
4. 判断每个功能模块属于A类（传统功能）还是B类（AI/异步/批处理功能），决定使用哪套要素模板
5. 生成完整文档大纲（含所有章节标题和层级），展示给用户确认

### 阶段三：逐章节生成

读取 [writing-guide.md](writing-guide.md) 获取写作规范，然后按以下顺序逐章生成：

**生成顺序：**
1. 一、概述（背景与建设目标）
2. 二、范围与边界
3. 三、总体业务流程概览
4. 四、系统功能（核心章节，逐个模块生成）
5. 五、非功能需求
6. 六、系统接口
7. 七、权限管理

**第四章生成规则（最重要）：**

对每个功能模块/专题：
- 先写 `业务目标`（3-5条）
- 再写 `业务场景`（2-4个典型场景，用"某角色在某情况下做某事"格式）
- 再写 `业务流程`（从触发到结束的完整链路）
- 最后逐个页面/操作写 `流程环节/功能模块`

对每个页面/操作，按以下要素输出：

**A类（传统功能）必填要素：**
```
【操作入口】：菜单路径或按钮位置
【功能描述】：一句话概述
【使用角色】：列出角色
【界面原型】：描述页面布局（查询区/列表区/表单区/按钮等）
【实现逻辑】：
  - 页面初始化逻辑
  - 按钮点击逻辑（含权限、校验、状态流转）
  - 数据交互逻辑
【字段说明】：表格形式列出每个字段的名称、类型、必填、可编辑、说明
```

**B类（AI/异步/批处理功能）额外要素（按需选用）：**
```
【交互过程】：流式状态提示的步骤文案（适用于AI推理、长时异步任务）
【提示词(Prompt)】：结构化Prompt框架（仅AI功能需要）
【语料/知识库】：AI依赖的知识范围和数据结构（仅AI功能需要）
【异步/批处理机制】：触发条件、执行策略、结果通知方式（仅异步/批处理功能需要）
【验证效果】：准确率等量化指标（AI功能）或成功率/吞吐量（批处理功能）
```

**涉及复杂关联时追加：**
```
【本体/知识支撑要求】：业务对象间的关联逻辑
【关联业务/接口】：上下游接口及影响
```

**附带结构化维度表：**
每个功能模块专题输出一个结构化维度总表（参考document-template.md中的维度表模板），汇总该模块的所有维度信息。

### 阶段四：质量检查与输出

执行以下检查：

1. **一致性检查**
   - 角色名称全文统一（不能一处写"管理员"另一处写"系统管理员"）
   - 状态名称全文统一（"生效"不能混写"已生效"）
   - 字段名称跨页面一致
   - 权限描述前后一致

2. **完整性检查**
   - 每个功能模块的A类6要素是否齐全
   - B类功能的额外要素是否补充（纯A类项目跳过此项）
   - 状态流转是否闭环（每个状态都有来源和去向）
   - 所有提到的按钮是否都有对应的逻辑描述

3. **流程闭环检查**
   - 数据从哪来、到哪去是否清晰
   - 各模块间的数据依赖关系是否正确
   - 反哺/闭环逻辑是否完整

4. **输出格式**
   - 默认输出为markdown格式，写入用户指定的文件
   - 如用户要求docx，使用python-docx生成
   - 文件名格式：`【项目名】系统功能需求说明书.md`

### 阶段五：迭代修订（按需进入）

用户对已生成文档提出修改意见后，进入本阶段：

1. **定位修改范围**：根据用户反馈定位到具体章节/模块
2. **局部重新生成**：仅重新生成受影响的章节，保持其余章节不变
3. **关联影响检查**：修改后检查是否影响其他章节的引用和一致性
   - 修改角色名 → 检查全文引用
   - 修改状态 → 检查状态流转图和相关按钮逻辑
   - 修改字段 → 检查所有涉及该字段的页面
4. **增量输出**：仅输出修改部分，标注改动位置

## 关键写作原则

1. **自顶向下**：先全局后局部，先概览后详情
2. **一致性优先**：同一概念全文用同一术语
3. **操作可执行**：每个功能描述必须具体到"谁在什么入口点什么按钮做什么事"
4. **状态驱动**：核心对象的行为围绕状态机描述
5. **边界清晰**：明确写出"做什么"和"不做什么"
6. **无冗余注释**：不写"本章节描述了..."之类的元描述，直接写内容

## 输出语言

- 默认使用中文
- 技术术语保持原文（如OCR、AI、PDF、API）
- 字段名中英文对照
